# 仓储机器人视觉导航系统验收清单

本文用于 Ubuntu 22.04 + ROS2 Humble 环境中的实机联调和课程演示验收。当前 macOS 环境只能运行静态测试，不能替代本清单中的 ROS/Gazebo 验证。

## 0. 本机静态检查

在进入 Ubuntu 联调前，先确认代码和文档没有明显静态问题：

```bash
cd ~/warehouse_ws/src/project
python3 tests/static_checks.py
```

成功标准：

- 10 项测试全部输出 `PASS`。
- `git status --short` 没有未提交的代码改动。

失败时优先检查：

- Python 文件是否有语法错误。
- README 是否仍记录当前 `bringup.launch.py mode:=sim|slam|nav` 流程。
- `package.xml` 是否声明了启动文件实际使用的依赖。

## 1. 编译验收

命令：

```bash
cd ~/warehouse_ws
source /opt/ros/humble/setup.bash
rosdep install --from-paths src --ignore-src -r -y
colcon build --symlink-install
source install/setup.bash
```

成功标准：

- `colcon build` 无报错。
- `source install/setup.bash` 后可以找到包：

```bash
ros2 pkg prefix project
```

失败时优先检查：

- 是否把项目放在 `~/warehouse_ws/src/project`。
- 是否执行了 `source /opt/ros/humble/setup.bash`。
- `package.xml` 中是否缺少 rosdep 可解析的依赖。
- 如果出现 `GLIBCXX` 错误，先执行 `conda deactivate`。

## 2. 仿真启动验收

命令：

```bash
cd ~/warehouse_ws
source /opt/ros/humble/setup.bash
source install/setup.bash
ros2 launch project bringup.launch.py mode:=sim
```

成功标准：

- Gazebo 能启动仓储世界。
- 机器人在 `x=-8.0, y=0.0` 附近生成。
- RViz 能打开导航配置。
- 动态叉车开始沿通道往返运动。

检查命令：

```bash
ros2 topic list
ros2 topic echo /clock --once
ros2 topic echo /scan --once
ros2 topic echo /odom --once
ros2 topic echo /imu --once
```

成功标准：

- `/clock` 持续发布仿真时间。
- `/scan` 有 `LaserScan` 数据。
- `/odom` 有里程计数据。
- `/imu` 有 IMU 数据。

失败时优先检查：

- Gazebo 是否成功加载 `warehouse.world`。
- 机器人是否完成 spawn。
- `robot_state_publisher` 是否拿到 `robot_description`。
- 服务器无图形界面时，是否可使用 Xvfb、VNC 或远程桌面。

## 3. TF 验收

命令：

```bash
ros2 run tf2_tools view_frames
ros2 run tf2_ros tf2_echo odom base_footprint
ros2 run tf2_ros tf2_echo base_link laser_link
ros2 run tf2_ros tf2_echo base_link camera_rgbd_link
```

成功标准：

- TF 链路包含 `odom -> base_footprint -> base_link`。
- `laser_link`、`camera_rgbd_link`、`imu_link` 挂在机器人基座下。
- `tf2_echo` 不持续报 extrapolation 或 frame 不存在。

失败时优先检查：

- `robot_state_publisher` 是否运行。
- `diff_drive` 插件是否发布 `odom -> base_footprint`。
- `robot.xacro` 中 link/joint 名称是否与配置一致。

## 4. 键盘遥控验收

新终端执行：

```bash
cd ~/warehouse_ws
source /opt/ros/humble/setup.bash
source install/setup.bash
ros2 run teleop_twist_keyboard teleop_twist_keyboard
```

检查命令：

```bash
ros2 topic echo /cmd_vel
ros2 topic echo /odom
```

成功标准：

- 按键后 `/cmd_vel` 有速度输出。
- 机器人在 Gazebo 中移动。
- `/odom` 位姿随运动变化。

失败时优先检查：

- 键盘遥控终端是否获得焦点。
- `diff_drive` 插件的 `command_topic` 是否为 `cmd_vel`。
- Gazebo 是否暂停。

## 5. 2D SLAM 建图验收

命令：

```bash
cd ~/warehouse_ws
source /opt/ros/humble/setup.bash
source install/setup.bash
ros2 launch project bringup.launch.py mode:=slam
```

检查命令：

```bash
ros2 topic echo /map --field header
ros2 topic echo /submap_list --once
ros2 topic echo /scan --once
ros2 topic echo /odom --once
```

成功标准：

- `/map` 持续更新。
- `/submap_list` 有 Cartographer 子图信息。
- 遥控机器人绕仓库一圈后，RViz 中货架和墙体轮廓清晰。
- 回到起点附近时地图没有明显撕裂或大幅错位。

失败时优先检查：

- `cartographer.lua` 中 `tracking_frame`、`published_frame`、`odom_frame` 是否与 TF 一致。
- `/scan` 是否正常发布。
- `/odom` 是否正常发布。
- 是否等待 Gazebo 完全启动后再开始建图。

## 6. 地图保存验收

命令：

```bash
mkdir -p ~/warehouse_ws/src/project/maps
ros2 run nav2_map_server map_saver_cli -f ~/warehouse_ws/src/project/maps/warehouse_map
```

成功标准：

- 生成 `maps/warehouse_map.yaml`。
- 生成 `maps/warehouse_map.pgm`。
- `warehouse_map.yaml` 中的 `image` 指向 `warehouse_map.pgm`。

检查命令：

```bash
ls -lh ~/warehouse_ws/src/project/maps
sed -n '1,40p' ~/warehouse_ws/src/project/maps/warehouse_map.yaml
```

失败时优先检查：

- `/map` 是否存在。
- map_saver 是否在已经完成建图后执行。
- `maps/` 目录是否有写入权限。

## 7. 导航启动验收

命令：

```bash
cd ~/warehouse_ws
source /opt/ros/humble/setup.bash
source install/setup.bash
ros2 launch project bringup.launch.py mode:=nav map:=~/warehouse_ws/src/project/maps/warehouse_map.yaml
```

成功标准：

- 如果地图存在，Nav2 正常启动。
- 如果地图不存在，launch 明确提示先建图并保存地图，然后退出。
- RViz 中能看到地图、机器人模型、LaserScan、global/local costmap。

检查命令：

```bash
ros2 lifecycle nodes
ros2 topic echo /map --once
ros2 topic echo /amcl_pose --once
ros2 action list | grep navigate
```

成功标准：

- `map_server`、`amcl` 和 Nav2 关键节点处于 active 状态。
- `/map` 能读取已保存地图。
- `/amcl_pose` 有定位输出。
- 存在 `navigate_to_pose` action。

失败时优先检查：

- `map:=...` 路径是否真实存在。
- 是否已经执行 `source install/setup.bash`。
- `nav2.yaml` 插件名称是否被当前 ROS2 Humble 支持。
- AMCL 的 `base_frame_id`、`odom_frame_id`、`global_frame_id` 是否与 TF 一致。

## 8. 多目标点导航验收

命令：

```bash
cd ~/warehouse_ws
source /opt/ros/humble/setup.bash
source install/setup.bash
ros2 run project waypoint_nav.py
```

成功标准：

- Nav2 action server 接受目标点。
- 机器人依次前往 LoadingZone A、ShelfCorridor B、LoadingZone C 和 Home。
- 遇到动态叉车时能减速、绕行或等待，不发生明显碰撞。
- 终端最终输出所有目标点完成。

检查命令：

```bash
ros2 topic echo /cmd_vel
ros2 topic echo /plan --once
ros2 topic echo /local_costmap/costmap --once
```

失败时优先检查：

- Nav2 是否已经 active。
- RViz 中是否已经设置初始位姿。
- 目标点坐标是否落在可通行区域。
- local costmap 是否接收 `/scan` 障碍物数据。

## 9. 3D 建图验收

命令：

```bash
cd ~/warehouse_ws
source /opt/ros/humble/setup.bash
source install/setup.bash
ros2 launch project rtabmap.launch.py
```

成功标准：

- RTAB-Map 节点正常启动。
- RGB、Depth、CameraInfo 和 Odom 输入能同步。
- RViz 或 rtabmap_viz 中能看到逐步生成的 3D 地图/点云。

检查命令：

```bash
ros2 topic echo /camera/rgb/image_raw --once
ros2 topic echo /camera/depth/image_raw --once
ros2 topic echo /camera/rgb/camera_info --once
ros2 topic list | grep rtabmap
```

失败时优先检查：

- 内存是否不足。
- RGB-D 相机话题名称是否与 `rtabmap.launch.py` remapping 一致。
- `/odom` 和 TF 是否可用。

## 10. 演示验收顺序

建议最终演示按以下顺序录制：

```text
1. 展示 Gazebo 仓储场景、机器人、货架和动态叉车
2. 展示 /scan、/odom、/imu 等传感器话题
3. 使用 mode:=slam 进行 2D 建图
4. 保存地图并展示 warehouse_map.yaml / warehouse_map.pgm
5. 使用 mode:=nav 加载地图和启动 AMCL + Nav2
6. 在 RViz 中展示地图、定位粒子、路径规划和 costmap
7. 运行 waypoint_nav.py 完成多目标点导航
8. 展示动态叉车避障效果
```

最终交付前确认：

- README 与实际运行命令一致。
- `python3 tests/static_checks.py` 通过。
- 地图文件已保存。
- 演示视频包含建图、定位、导航和动态避障。
- 技术报告中说明当前项目使用 `bringup.launch.py mode:=sim|slam|nav`。
